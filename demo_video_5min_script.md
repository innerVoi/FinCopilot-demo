# FinCopilot V2.2 5 分钟 Demo 视频演示要点与讲稿

## 视频目标

在 5 分钟内说明 FinCopilot V2.2 解决什么问题、Demo 已经实现什么、用户如何完成一次财务分析，以及为什么 V2.2 相比普通记账软件更适合校园周边小微商家。

建议录屏前准备：

- 启动 Streamlit 应用，默认进入 `Copilot 主界面`。
- 使用默认样例数据，避免现场上传文件耗时。
- 如果没有配置 API Key，保留 fallback 状态即可；这是 V2.2 的设计亮点之一。
- 推荐只演示一个主问题：`未来 30 天现金流安全吗？`

## 0:00-0:30 开场：项目定位

画面动作：

- 展示应用首页或 README 标题。
- 镜头停留在 `FinCopilot V2.2` 标题和 Copilot 主界面。

讲稿：

> FinCopilot V2.2，是一个面向校园周边小微商家的 AI 财务决策 Copilot。  
> 它的目标不是替代专业财务顾问，而是帮助小饭馆、奶茶店、咖啡店、打印店、便利店这类缺乏专业财务的小商家，用一个简单的 Chat 窗口看懂经营数据。  
> 这些商家通常有微信支付宝流水、外卖账单、进货记录和发票，但问题是数据分散，也不一定懂财务分析。作为一个demo演示，FinCopilot假设数据已可用的情况下，重点回答三个关键问题：钱够不够、哪些支出有问题、下一步应该采取什么行动。

## 0:30-1:10 展示 V2.2 页面结构

画面动作：

- 展示左侧 sidebar。
- 依次指向四个入口：`Copilot 主界面`、`分析详情`、`行动与报告`、`数据与设置`。
- 回到 `Copilot 主界面`。

讲稿：

> 第一个是 Copilot 主界面，用来上传数据、提问、看结果卡片和详细预览。  
> 第二个是分析详情，用来查看预算、发票、现金流、异常支出和 Agent 执行轨迹。  
> 第三个是行动与报告，用来沉淀本轮分析产生的行动项和报告。  
> 第四个是数据与设置，用来查看数据预览、字段说明、模型状态和安全边界。  
> 对普通商家来说，最重要的是第一个页面：不用学习复杂系统，直接提问就可以。

## 1:10-1:50 展示数据状态、上传与 fallback

画面动作：

- 在 Copilot 主界面展示数据状态卡片。
- 展示快速上传区域或默认样例数据提示。
- 展示 Agent 状态提示，尤其是 fallback 提示。

讲稿：

> 这里可以看到当前交易、发票和财务目标的数据状态。实际使用时，商家可以上传 CSV，比如收款流水、发票记录和利润目标。  
> 为了方便演示，系统采用了生成的模拟样例数据，以保证在缺乏真实商家数据的情况下，能够跑通完整流程。  
> 分析方法，系统默认接入api以实现多agent流程，由一个Manager Agent进行意图识别和规划，然后调用不同的Specialist Agents去执行对应设定的功能。
> 当api_key无法使用时，系统自动切换为fallback模式，利用静态模板+数据抽取返回输出结果，避免直接报错而无法使用。
> 这样的整体设计对新手用户很重要，这允许他们能够在不理解技术配置，通过自然语言输入获得预期的财务分析结果。
> 此外，fallback模式的注入防止了api调用失败时系统直接奔溃的潜在风险，在能为商家提供基础结果的同时，给系统后台调整也保障了时间。

## 1:50-2:45 核心演示：运行现金流分析

画面动作：

- 点击推荐任务 `未来 30 天现金流安全吗？`，或在 Chat 输入框输入同一句。
- 等待结果生成。
- 展示结果卡片区域。

讲稿：

> 现在我演示一个小商家最常见的问题：未来 30 天现金流安全吗？  
> 用户可以点击推荐任务，也可以直接在 Chat 里输入这个问题。  
> 提交后，系统会先由 Manager Agent 理解任务，判断这是现金流风险问题；然后调用本地工具读取预算、发票和现金流摘要；再由 Cashflow Agent 和 Report Agent 生成解释；最后经过 Safety Guardrails，生成安全的回复、行动项和报告。  
> 如果当前是真实 Agent 模式，这里会显示 Agent 分析结果；如果是 fallback 模式，也会基于本地确定性工具给出可用分析。

## 2:45-3:35 展示结果：从数据到行动

画面动作：

- 展示 headline、summary、关键指标卡片。
- 展示风险卡片、建议行动、补充问题。
- 展开详细结果预览。

讲稿：

> 分析完成后，用户首先看到的是卡片化结果，而不是一大段技术 JSON。  
> 这里会展示现金流结论、关键指标，比如当前余额估算、未来 30 天待付发票、预计余额和风险等级。  
> 如果存在风险，系统会解释风险原因，比如近期有大额付款、发票集中到期，或者经营现金流不足。  
> 更重要的是，它会生成建议行动，例如优先跟进应收款、核查近期到期发票、推迟非必要采购，或者补充真实账户余额。  
> 展开详细预览后，可以看到现金流、发票、异常支出、目标进度、行动项、报告和 Agent 执行摘要。这样用户在一个页面里就能从“我问了一个问题”走到“我知道下一步该做什么”。

## 3:35-4:20 展示分析详情与行动报告

画面动作：

- 切换到 `分析详情`。
- 快速展示预算与分类、发票与现金流、异常支出、财务目标、Agent 执行轨迹。
- 切换到 `行动与报告`。
- 展示行动项、报告和 Trace。

讲稿：

> 如果用户想看证据，可以进入分析详情。  
> 这里保留了完整的预算分类、发票与现金流、异常支出识别、财务目标分析，以及 Agent 执行轨迹。  
> 这层页面不是给用户增加操作负担，而是在用户需要核查时提供依据。  
> 接着看行动与报告。FinCopilot V2.2 不只是聊天回复，而是会把建议同步成行动项，并生成 Multi-Agent 报告和 Trace。  
> 该设计适合小商家每周复盘：哪些付款要处理、哪些支出要核查、哪些目标有风险，都可以沉淀下来作为知识，并在后续设计中通过人在回路+RAG的方式，将知识和人类审查的反馈保存下来，成为用户的私域经验，从而增强模型的分析和判断能力，不断提升用户的产品使用体验。

## 4:20-4:50 展示数据与安全边界

画面动作：

- 切换到 `数据与设置`。
- 展示数据预览、Agent 与模型状态、安全边界 Tab。

讲稿：

> 最后看数据与设置页。这里可以检查上传数据、字段说明、Agent 与模型状态。  
> V2.2 也明确保留安全边界：它只做财务整理、风险提醒和教育性支持，不提供投资、税务、法律或债务处置建议，也不会自动付款或转账。  
> 对小微商家来说，这个边界很重要：AI 可以帮忙发现问题和整理行动，但关键决策仍然要核查原始凭证。

## 4:50-5:00 收尾总结

画面动作：

- 回到 `Copilot 主界面` 或停留在结果卡片。

讲稿：

> 总结一下，FinCopilot V2.2 的价值是：用一个统一 Chat 窗口，把离散的经营数据转化为现金流判断、异常识别和行动建议，以简单易上手的形式，帮助小微商家快速、有效的实现财务分析和行动规划。  
> 相比普通记账软件，它更强调数据分析、解释和决策支持；相比复杂财务系统，它更适合校园周边小微商家低成本试用和快速验证。在后续的版本中，我们计划进一步补充：高效的多源收支数据处理、人在回路的异常审查反馈、基于RAG的知识沉淀和经营策略推荐等关键功能。也期待有一天FinCopilot能从demo走向一个落地的产品。

## 录制时的重点提示

- 不要演示太多问题，5 分钟内只跑一个核心问题最稳。
- 重点展示“统一 Chat 窗口 + 默认 Agent + 自动 fallback + 结果卡片 + 行动报告”。
- 如果现场结果生成较慢，可以提前跑一次，让页面已有结果，再录制讲解。
- 不需要解释代码实现细节，除非课程要求技术说明。
- 遇到 fallback 状态不要避开，直接把它讲成产品鲁棒性设计。

## 可选备用问题

如果现金流问题不适合当前数据，可以换成：

1. `这个月哪些支出最可疑？`
2. `有哪些发票或付款要优先处理？`
3. `帮我生成本周财务行动清单。`

## 一句话版本

FinCopilot V2.2 是一个面向校园周边小微商家的 AI 财务决策 Copilot：商家在一个 Chat 主界面上传数据并提问，系统通过 Agent 和本地工具完成现金流、异常支出、发票和目标分析，即使 Agent API 不可用也会自动 fallback，最终输出可理解的风险解释、行动项和经营报告。
